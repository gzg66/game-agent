# 冷启动探索流程设计

## 1. 文档目的

本文档用于定义 AI 手游测试系统中的“冷启动探索”流程，指导后续开发者和 Cursor 在没有完整先验知识的情况下，如何通过可控方式建立初始世界模型。

冷启动探索不是为了“一次采完全部 UI 树”，而是为了在尽量低风险、可解释、可持续补全的前提下，快速建立：

- 初始页面库
- 初始控件语义库
- 初始状态转移图
- 初始模块边界
- 初始异常基线

本文档应与以下文档配合使用：

- `AI手游自主游玩测试整体技术方案.md`
- `Agent单步决策协议.md`
- `世界模型数据结构设计.md`

## 2. 冷启动探索的定位

冷启动探索是自主游玩 Agent 上线之前的准备阶段，也是后续长期运行中的补全机制。

其目标不是通关游戏，而是尽快回答下面这些问题：

- 当前项目里有哪些主要页面
- 每个页面有哪些核心可交互对象
- 哪些按钮大概率用于推进主流程
- 哪些按钮大概率是关闭、返回、确认、领取
- 哪些页面属于大厅、弹窗、新手、战斗、奖励、商店等模块
- 从一个页面点击某个动作后，通常会跳到哪里
- 哪些页面或动作可能高风险

## 3. 核心原则

### 3.1 优先建立初始世界模型，不追求一次全覆盖

游戏中很多页面是条件触发的，例如：

- 等级解锁
- 主线进度解锁
- 资源充足时出现
- 活动开放时出现
- 战斗结束后出现
- 某个弹窗关闭后出现

因此冷启动阶段不能把“全量覆盖”作为强目标。

### 3.2 优先低风险探索

冷启动阶段的动作优先级应偏保守，优先探索：

- 开始
- 下一步
- 进入
- 确认
- 关闭
- 返回
- 跳过
- 领取

应显著降权或阻断：

- 充值
- 支付
- 购买
- 删除账号
- 跳转外链
- 明显付费入口

### 3.3 页面级建模优先于动作级穷举

冷启动阶段的主要目标是识别页面类别、页面差异和页面关系，而不是机械地把每个按钮都点一遍。

### 3.4 每次探索都必须留下结构化证据

至少需要记录：

- 页面快照
- 候选动作
- 实际动作
- 页面变化
- 异常信号

## 4. 冷启动探索的输入与输出

## 4.1 输入

冷启动探索依赖以下基础能力：

- 可以稳定获取 Poco UI 树
- 可以稳定截图
- 可以执行点击、返回、关闭、确认等基础动作
- 可以采集日志
- 可以检测进程状态

## 4.2 输出

冷启动探索完成后，至少应输出：

- 初始 `PageObservation` 集合
- 初始 `PageState` 集合
- 初始 `SemanticNode` 集合
- 初始 `CandidateAction` 统计
- 初始 `StateTransition` 图
- 冷启动探索报告

## 5. 整体流程

冷启动探索建议分为六步：

1. 运行前准备
2. 首屏采集
3. 页面聚类与初始语义识别
4. 受控动作探索
5. 状态图沉淀
6. 冷启动结果验收

## 5.1 运行前准备

### 目标

确保进入冷启动时，环境尽量干净可重复。

### 建议步骤

1. 重启游戏进程
2. 清理临时弹窗和已有悬浮层
3. 清空或切分本轮日志窗口
4. 记录版本号、设备信息、启动时间
5. 获取首次稳定截图和 UI 树

### 输出

- `RunContext`
- 第一份 `PageObservation`

## 5.2 首屏采集

### 目标

在不做复杂动作前，先稳定拿到初始页面信息。

### 必须采集

- 原始 UI 树
- 截图
- 根节点
- 可点击节点
- 文本节点
- 页面签名
- 页面停留时长
- 首屏日志片段

### 必须判断

- 当前是否是登录页
- 当前是否存在弹窗
- 当前是否在大厅
- 当前是否在新手引导
- 当前是否有高风险入口

## 5.3 页面聚类与初始语义识别

### 目标

从原始观测中形成“页面状态”，而不是只存原始快照。

### 页面聚类建议依据

- 页面签名
- 根节点结构
- 关键文本
- 可点击节点集合
- 页面标签规则

### 初始页面语义识别建议优先做的类别

- `login`
- `dialog`
- `lobby`
- `guide`
- `reward`
- `battle_prepare`
- `battle_running`
- `battle_result`
- `shop`
- `unknown`

### 初始控件语义识别建议优先做的角色

- `close`
- `back`
- `confirm`
- `skip`
- `reward_claim`
- `primary_entry`
- `battle_start`
- `dangerous_action`

## 5.4 受控动作探索

### 目标

通过低风险、少量、高价值动作逐步建立页面转移关系。

### 受控动作探索的基本策略

1. 每个页面只优先尝试有限个高价值候选动作
2. 每次动作后必须重新观测页面
3. 如果页面变化明显，则沉淀一条状态转移边
4. 如果页面未变化，则降低该动作后续优先级
5. 遇到新页面时优先做页面建模，不要立刻继续乱点

### 建议的动作优先级

第一优先级：

- 关闭弹窗
- 确认
- 下一步
- 开始
- 进入
- 领取

第二优先级：

- 返回
- 跳过
- 功能入口
- 战斗入口

第三优先级：

- 非核心装饰按钮
- 商业化入口
- 不明确用途入口

### 单页动作数量控制

建议：

- 每个页面冷启动阶段优先探索 3 到 8 个动作
- 不要对同页面所有按钮做穷举点击

## 5.5 状态图沉淀

### 目标

把页面和动作组织成后续 Agent 可利用的“初始地图”。

### 至少要沉淀的关系

- 页面到页面的跳转
- 哪个动作触发了跳转
- 该动作成功率
- 该动作所属模块
- 该动作是否高风险
- 页面是否有回环

### 需要额外标记的页面

- 高频出现页面
- 弹窗类页面
- 阻塞风险页面
- 商业化页面
- 战斗相关页面
- 奖励相关页面

## 5.6 冷启动结果验收

### 验收标准不应是

- 点完所有按钮
- 覆盖全部页面
- 一次跑完整个游戏

### 验收标准应是

- 形成一批稳定页面签名
- 识别出主要模块入口
- 建立初始状态图
- 沉淀一批基础控件语义
- 建立一批高价值候选动作模板
- 能为后续 Agent 提供足够的先验

## 6. 冷启动探索的分层实现建议

## 6.1 观测层

职责：

- 采树
- 截图
- 抽取可点击节点
- 抽取文本节点
- 生成页面签名

输出：

- `PageObservation`

## 6.2 语义层

职责：

- 给页面打类别标签
- 给控件打语义角色
- 标记风险节点

输出：

- `PageState`
- `SemanticNode`

## 6.3 探索层

职责：

- 从当前页面生成候选动作
- 选择要尝试的动作
- 控制每页的探索深度
- 控制回退和继续策略

输出：

- `CandidateAction`
- `ActionExecution`

## 6.4 图谱层

职责：

- 沉淀状态图
- 沉淀页面归并结果
- 沉淀成功和失败动作统计

输出：

- `StateTransition`
- `StateGraphMemory`

## 7. 冷启动阶段的停止条件

满足以下任一条件可以停止：

- 达到最大探索步数
- 达到最大页面数
- 连续多步无新增页面
- 连续多步无高价值新转移
- 命中严重异常
- 进入明显高风险区域

## 8. 冷启动与自主游玩的关系

冷启动探索不是独立于自主游玩的单独系统，而是自主游玩前期和运行中的一个子机制。

关系如下：

- 冷启动探索负责建立初始世界模型
- 自主游玩 Agent 负责利用世界模型推进流程
- 自主游玩过程中又会继续产生新的冷启动式补全数据

因此实现上建议：

- 不要单独做一套完全不同的冷启动代码
- 应尽量复用相同的数据结构和观测能力

## 9. 开发约束

后续实现冷启动探索时，必须遵守：

1. 不要把冷启动做成“穷举点所有按钮”的遍历器。
2. 不要只存原始 UI 树，不做页面和控件语义化。
3. 不要为了提高覆盖率而过早进入高风险区域。
4. 不要把探索逻辑写死在单项目 runner 中。
5. 不要在没有状态图沉淀的情况下直接进入 LLM 决策。

## 10. 推荐的开发顺序

建议按以下顺序开发：

1. 页面快照采集
2. 页面签名与页面归并
3. 控件语义识别
4. 候选动作生成
5. 单页受控探索
6. 状态图沉淀
7. 冷启动探索报告

## 11. 冷启动报告建议内容

冷启动探索结束后，建议输出以下内容：

- 新发现页面数
- 已归并页面数
- 主要模块入口
- 高频控件语义标签
- 高价值转移路径
- 高风险页面与动作
- 冷启动阶段异常
- 建议进入自主游玩的初始目标

## 12. 一句话总结

冷启动探索的本质不是“先乱点一遍”，而是“用低风险、结构化、可回放的方式，为 AI 建立第一版可用世界模型”。
