# 异常检测器设计

## 1. 文档目的

本文档定义 AI 手游测试系统中的异常检测器设计，用于规范问题发现、异常信号建模、问题归并、证据保存和报告输出方式。

目标是让系统在运行过程中，而不是只在结束后，持续发现并记录以下异常：

- 崩溃
- 阻塞
- 项目报错
- 数值错误
- 资源缺失

本文档默认与以下文档配合使用：

- `AI手游自主游玩测试整体技术方案.md`
- `Agent单步决策协议.md`
- `世界模型数据结构设计.md`
- `冷启动探索流程设计.md`

## 2. 异常检测器的定位

异常检测器不只是日志扫描器，也不是纯粹的后处理分析模块。

它应当是运行时系统中的常驻能力，参与每一步：

- 动作前检查
- 动作后检查
- 页面停留检查
- 模块切换检查
- 整轮结束检查

异常检测器要回答的不只是“有没有问题”，还要回答：

- 这是什么类型的问题
- 它发生在什么页面
- 可能是由什么动作触发的
- 有哪些证据支持这个判断
- 是否足以形成正式问题记录

## 3. 核心原则

### 3.1 信号与问题分层

必须区分：

- `AnomalySignal`：轻量级异常信号
- `IssueRecord`：正式问题记录

不要把所有可疑信号都直接当作正式 bug。

### 3.2 运行时持续检测

异常检测必须嵌入执行闭环，不能只在跑完后统一分析。

### 3.3 证据优先

每个正式问题至少要保留一部分证据：

- 页面信息
- 动作链
- 时间点
- 相关日志
- 截图或页面摘要

### 3.4 分层置信度

不是所有异常都能做到强确定性，因此需要对问题和信号都标注置信度或等级。

### 3.5 先做稳定问题，再做复杂问题

优先保证以下三类稳定：

- 崩溃
- 阻塞
- 项目报错

数值错误和资源缺失先做弱检测，逐步增强。

## 4. 检测对象分层

建议把异常检测器拆成三层：

1. 信号采集层
2. 问题判定层
3. 问题归并与输出层

## 4.1 信号采集层

负责收集运行时异常线索。

信号来源包括：

- UI 树变化
- 页面停留时间
- 动作成功率
- 页面回环率
- 截图特征
- 客户端日志
- 进程状态
- 设备状态

## 4.2 问题判定层

负责根据多类信号综合判断，决定是否生成正式问题记录。

## 4.3 问题归并与输出层

负责：

- 去重
- 合并重复问题
- 补充复现路径
- 输出到报告和持久化存储

## 5. 异常类型设计

## 5.1 崩溃

### 定义

运行过程中的非预期退出、闪退、崩溃、进程异常终止。

### 主要信号

- 进程不存在
- Poco 断连
- 回到桌面
- 黑屏或启动页长时间停留
- 日志中出现 crash、fatal、tombstone、SIGABRT、SIGSEGV

### 高置信度触发条件

满足以下任一即可生成正式问题：

- 进程退出且无法自动恢复
- 日志命中明确 crash 特征
- Poco 连接丢失且游戏进程消失

### 必须保留的证据

- 发生时间
- 当前页面
- 最近动作链
- 日志片段
- 截图
- 进程状态

## 5.2 阻塞

### 定义

测试无法继续推进，但客户端并未崩溃。

### 主要信号

- 页面长时间不变化
- 连续点击无效果
- 在少量页面间反复循环
- 按钮越来越少但目标没有推进
- 多次进入相同状态签名

### 阻塞检测不能只看单一阈值

建议结合：

- 页面停留时长
- 连续无变化步数
- 动作成功率
- 状态图回环率
- 目标推进率

### 建议分级

- `low`：短时可疑停留
- `medium`：连续多步无推进
- `high`：确认流程无法推进

### 必须保留的证据

- 页面签名
- 最近动作链
- 页面停留时间
- 最近若干步页面变化情况

## 5.3 项目报错

### 定义

客户端或项目日志中出现明确的报错、异常、断言、严重告警。

### 主要信号

- `error`
- `exception`
- `assert`
- `fatal`
- 项目自定义错误关键字

### 高置信度触发条件

- 错误关键字与明确异常栈同时出现
- 同一时段多条严重错误日志集中出现
- 错误日志与页面异常或动作失败同时出现

### 注意事项

- 不能只按关键字做最终判定
- 需要与时间窗口、页面、动作链绑定

### 必须保留的证据

- 日志原文
- 命中关键字
- 当前页面
- 最近动作链
- 日志时间窗口

## 5.4 数值错误

### 定义

游戏中的关键奖励、资源、结算数值明显不合理。

### 早期阶段建议做弱检测

优先关注：

- 负数
- `NaN`
- `Inf`
- 极端异常大数
- 奖励前后变化明显异常
- 结算结果与文案语义明显不一致

### 后续增强方向

- 引入项目规则知识
- 引入资源变化预期模型
- 引入战斗结算比对逻辑

### 必须保留的证据

- 数值出现页面
- 原始文本
- 前后页面变化
- 最近动作

## 5.5 资源缺失

### 定义

图片、文本、UI 资源显示异常或明显未替换。

### 早期阶段建议做弱检测

关注以下信号：

- 文本空白
- missing / default / null / todo / placeholder
- 按钮存在但无有效文本或图像
- 截图中出现大面积白块或占位图

### 后续增强方向

- OCR 校验
- 图像模型检测
- 图文一致性比对

### 必须保留的证据

- 页面截图
- 页面语义信息
- 可疑节点文本
- 当前页面签名

## 6. 异常信号对象设计

建议定义 `AnomalySignal`

建议字段：

- `signal_id`
- `signal_type`
- `source`
- `severity_hint`
- `score`
- `page_signature`
- `page_name`
- `step_index`
- `evidence`
- `captured_at`

### `signal_type` 建议枚举

- `process_exit`
- `disconnect`
- `long_stay`
- `repeat_state`
- `action_no_effect`
- `log_error`
- `suspicious_number`
- `missing_resource`

## 7. 正式问题对象设计

建议使用统一的 `IssueRecord`

建议字段：

- `issue_id`
- `category`
- `severity`
- `confidence`
- `title`
- `page_signature`
- `page_name`
- `module_name`
- `action_index`
- `action_label`
- `reproduction_path`
- `related_logs`
- `related_screenshots`
- `details`
- `occurred_at`

## 8. 检测时机设计

异常检测建议挂在以下时机：

1. 动作前
2. 动作后
3. 页面刷新后
4. 停留超时后
5. 模块切换前后
6. 运行结束前

### 动作前检测

主要用于：

- 检查进程状态
- 检查已有严重异常
- 检查页面是否已明显阻塞

### 动作后检测

主要用于：

- 检查动作是否无效
- 检查页面是否发生异常变化
- 检查是否伴随错误日志

### 停留超时检测

主要用于：

- 检测阻塞
- 检测加载卡死
- 检测页面循环

## 9. 问题判定策略

## 9.1 单信号直接判定

只有少数情况允许单信号直接生成正式问题：

- 明确 crash 日志
- 进程消失
- Poco 断连且进程退出

## 9.2 多信号组合判定

大多数异常应采用组合判定，例如：

- `long_stay + repeat_state + action_no_effect` -> 阻塞
- `log_error + action_failed + suspicious_page_change` -> 项目报错
- `suspicious_number + settlement_page + reward_change_abnormal` -> 数值错误
- `missing_resource + screenshot_abnormal + empty_text` -> 资源缺失

## 9.3 问题置信度建议

建议分为：

- `high`
- `medium`
- `low`

`high` 表示可直接进入报告主问题区。  
`medium` 表示需要人工关注。  
`low` 表示暂存为疑似问题。

## 10. 去重与归并策略

不要让同一问题在长时运行中无限重复报出。

### 建议归并键

可综合以下字段：

- `category`
- `page_signature`
- `action_label`
- `error_fingerprint`

### 合并后应保留

- 首次发现时间
- 最近发现时间
- 发生次数
- 最近复现路径
- 最新日志片段

## 11. 与状态图和 Agent 的联动

异常检测器不应孤立工作。

### 与状态图联动

- 页面回环率可用于判定阻塞
- 某些转移边可标记为高风险边
- 某些页面可标记为历史异常高发页面

### 与 Agent 联动

异常检测结果可影响下一步决策：

- 命中高风险异常时停止当前路径
- 命中可疑阻塞时优先回退
- 命中项目报错时保留现场
- 命中资源缺失时增加截图证据

## 12. 报告输出要求

每个正式问题输出时，建议至少包含：

- 问题类型
- 问题级别
- 发生时间
- 所在页面
- 最近动作链
- 主要证据
- AI 影响描述

对于稳定问题，还建议输出：

- 触发次数
- 高频触发路径
- 是否自动恢复

## 13. 开发顺序建议

建议按以下顺序实现：

1. 崩溃检测
2. 阻塞检测
3. 项目报错检测
4. 问题去重归并
5. 数值错误弱检测
6. 资源缺失弱检测
7. 图像与 OCR 增强检测

## 14. 开发约束

后续实现异常检测器时，必须遵守：

1. 不要只扫描日志，不结合页面和动作上下文。
2. 不要把所有可疑情况都直接升级为正式问题。
3. 不要只在整轮运行结束后再分析异常。
4. 不要让同一问题在长时运行中无限重复输出。
5. 不要在没有证据的情况下给出高置信度结论。

## 15. 最小可落地版本建议

第一版建议至少完成：

- 进程崩溃检测
- Poco 断连检测
- 长时间停留检测
- 连续无效点击检测
- 日志报错检测
- 统一 `IssueRecord` 输出

只要这部分稳定，就足以支撑长时巡航的第一版问题发现能力。

## 16. 一句话总结

异常检测器不是“跑完以后扫日志”，而是“贯穿观察、执行、验证全过程的实时问题发现系统”。
