# AI 手游自主游玩测试整体技术方案

## 1. 文档目的

本文档用于统一后续开发方向，约束 Cursor 和人工开发者在实现 AI 手游自动测试能力时的技术路线。

目标不是继续把系统做成“规则驱动的 UI 遍历器”，而是建设一套：

- 能持续观察游戏当前状态的运行时系统
- 能逐步建立游戏世界模型的认知系统
- 能基于候选动作和历史记忆进行自主决策的 Agent 系统
- 能在游玩过程中发现问题并在结束后自动总结的测试系统

后续开发时，默认优先遵循本文档。若与已有代码实现冲突，以本文档描述的目标架构为准，再决定是否重构现有实现。

## 2. 核心目标

### 2.1 总体目标

构建一套 AI 驱动的手游自动测试系统，使 AI 能够像玩家一样理解界面、推进流程、探索模块、发现异常并输出总结报告。

### 2.2 明确边界

本项目的目标不是：

- 只靠一次运行拿到“全部 UI 树”
- 只用固定规则或关键字去遍历页面
- 只输出动作流水式报告
- 把 LLM 当成裸控脚本，直接从全量 UI 树自由选点

本项目的目标是：

- 先通过冷启动探索建立初始世界模型
- 在自主游玩过程中持续补全世界模型
- 让系统先生成候选动作，再由 AI 在候选动作中选择
- 把异常检测嵌入运行过程，而不是只做事后分析
- 最终输出覆盖、路径、异常、结论都完整的结构化报告

## 3. 总体设计原则

### 3.1 认知优先，不是遍历优先

系统首先要回答“当前页面是什么”“有哪些可交互对象”“各对象可能有什么作用”，然后才是决定“下一步做什么”。

### 3.2 世界模型持续更新

世界模型不是一次构建完成，而是在每轮测试中不断补全。

### 3.3 候选动作先生成，AI 后选择

不允许把全量 UI 树原文直接交给模型让其自由发挥。必须先由系统层做候选动作筛选、语义压缩、风险过滤，再让 AI 决策。

### 3.4 异常检测嵌入决策闭环

每次观察、决策、执行之后都要检查是否出现崩溃、阻塞、项目报错、数值异常、资源异常。

### 3.5 报告服务于研发定位

报告不只是动作记录，而要包含：

- 覆盖情况
- 关键路径
- 问题分类
- 复现步骤
- 相关日志
- 影响描述
- AI 结论

## 4. 目标架构

整体分为五层：

1. 观测层
2. 世界模型层
3. 候选动作层
4. Agent 决策层
5. 异常检测与报告层

### 4.1 观测层

职责：

- 采集 UI 树
- 采集截图
- 采集日志
- 采集页面停留时长
- 采集动作前后状态差异
- 采集运行环境信息

输入：

- Poco UI 树
- 游戏截图
- adb / 客户端日志
- 设备状态

输出：

- 标准化页面快照 `PageObservation`
- 动作执行记录 `ActionExecution`
- 日志片段 `RuntimeLogSlice`

### 4.2 世界模型层

职责：

- 识别页面
- 理解控件语义
- 构建状态图
- 记录页面转移关系
- 记录历史成功动作和失败动作
- 为不同版本和不同项目沉淀可复用的先验知识

输出：

- 页面节点库
- 页面签名库
- 控件语义库
- 状态转移图
- 模块标签
- 风险标签

### 4.3 候选动作层

职责：

- 从当前页面中抽取可点击对象
- 基于历史经验补充推荐动作
- 过滤危险动作
- 将原始控件压缩为带语义的候选动作

输出：

- `CandidateAction[]`

### 4.4 Agent 决策层

职责：

- 判断当前页面属于什么模块
- 判断当前阶段目标是什么
- 从候选动作中选择最像人会做的下一步
- 决策失败后重规划
- 保留决策理由，支持回放和分析

### 4.5 异常检测与报告层

职责：

- 在执行过程中检测异常
- 输出结构化问题记录
- 汇总覆盖与路径
- 输出 AI 总结报告

## 5. 分阶段技术流程

## 5.1 第一阶段：冷启动探索

### 目标

快速建立初始世界模型，而不是追求一次获取全部 UI 树。

### 必须采集的数据

每个页面至少采集：

- 页面签名
- 页面截图
- UI 树全量节点
- 可点击节点列表
- 文本节点列表
- 根节点信息
- 页面出现时间
- 页面停留时长

每次动作至少记录：

- 动作前页面签名
- 动作后页面签名
- 点击目标
- 动作语义
- 执行耗时
- 是否成功
- 页面变化摘要

### 输出产物

- 初始状态图
- 初始页面库
- 初始控件语义库
- 初始高频跳转关系

### 注意事项

- 不以“全量覆盖”为验收标准
- 以“是否足以支撑下一阶段 Agent 决策”为验收标准

## 5.2 第二阶段：页面与控件语义化

### 目标

让系统知道页面和按钮“是什么”，而不是只知道“它存在且可点击”。

### 页面语义标签示例

- 登录页
- 大厅
- 角色页
- 背包
- 活动页
- 商店
- 战斗中
- 战斗结算
- 新手引导
- 弹窗

### 控件语义标签示例

- 进入主流程
- 关闭弹窗
- 返回上一级
- 领取奖励
- 确认
- 跳过
- 战斗开始
- 战斗结算确认
- 功能入口
- 付费相关
- 危险动作

### 实现原则

- 先用规则做基础识别
- 再用 AI 做语义补充和归纳
- 语义结果必须可缓存、可复用、可覆盖修正

## 5.3 第三阶段：候选动作生成

### 目标

由系统先生成一组高质量候选动作，再交给 AI 选择。

### 候选动作来源

- 当前页面所有显式可点击节点
- 历史上该页面常见成功动作
- 模块内常见推进动作
- 通用动作模板

### 通用动作模板示例

- 关闭
- 返回
- 领取
- 开始
- 下一步
- 跳过
- 确认
- 再次挑战
- 结算确认

### 候选动作过滤原则

- 过滤危险动作
- 降低商业化入口优先级
- 降低重复失败动作优先级
- 降低已经验证无收益动作优先级
- 保留少量探索性动作，避免陷入局部最优

## 5.4 第四阶段：AI Agent 自主游玩

### 目标

让 AI 在候选动作集合中做出像人一样的决策。

### 单步决策循环

1. 观察当前页面
2. 识别页面与模块
3. 读取历史记忆
4. 生成候选动作
5. 对候选动作做语义压缩
6. AI 选择下一步动作
7. 执行动作
8. 对比执行前后差异
9. 判断是否成功推进
10. 更新世界模型与历史记忆
11. 检查是否出现异常
12. 决定继续、回退、重试或切换目标

### Agent 输入

每一步输入至少包含：

- 当前页面语义标签
- 当前页面摘要
- 最近若干步动作及结果
- 当前候选动作列表
- 当前测试目标
- 风险动作提示
- 历史成功经验
- 当前异常信号

### Agent 输出

每一步输出至少包含：

- 选择的动作
- 选择原因
- 预期结果
- 失败后的备选方案
- 当前对页面和模块的判断

### 实现要求

- 必须保留决策理由
- 必须支持回放
- 必须支持失败重规划

## 5.5 第五阶段：异常检测

### 原则

异常检测必须嵌入运行过程中，而不是跑完以后再看。

### 异常类型

#### 崩溃

检测信号：

- 进程退出
- Poco 断连
- 回到桌面
- 黑屏或启动页异常停留
- 日志出现 crash / fatal / tombstone

要求：

- 稳定输出问题记录
- 保留最近动作链
- 保留日志和时间点

#### 阻塞

检测信号：

- 页面长时间不变化
- 多次点击无响应
- 在少量页面间循环
- 可点击动作不断减少但流程无推进

要求：

- 同时结合停留时长、页面变化率、动作成功率、状态回环率判断
- 不能只用单一阈值判断

#### 项目报错

检测信号：

- 日志中出现 `error`、`exception`、`assert`、`fatal`
- 客户端可视错误面板
- 项目自定义错误关键字

要求：

- 绑定到当前页面和最近动作链
- 形成可复现问题记录

#### 数值错误

第一阶段先做弱检测：

- 负数
- `NaN`
- `Inf`
- 明显异常大数
- 前后奖励变化异常
- 结算数据与文案不一致

要求：

- 前期不承诺强校验
- 后续再逐步接入项目知识和规则

#### 资源缺失

第一阶段先做弱检测：

- 文本空白
- 占位资源
- missing / default / null / todo
- 明显按钮区域无有效资源
- 截图中白块、缺图、未替换资源

要求：

- UI 树与截图信号结合判断

## 5.6 第六阶段：总结报告

### 报告必须包含的内容

#### 覆盖层

- 覆盖页面数
- 覆盖关键页面
- 覆盖关键路径
- 覆盖模块列表
- 未覆盖核心模块

#### 行为层

- AI 做过的关键决策
- 高成功率动作
- 高失败率动作
- 重试与回退情况

#### 异常层

- 问题分类
- 问题级别
- 触发页面
- 触发动作
- 最近路径
- 关联日志
- 影响描述

#### 总结层

- 本轮测试结论
- 稳定性判断
- 风险最高模块
- 推荐补充探索方向
- 疑似根因

## 6. 数据模型建议

以下为后续开发时建议优先抽象的数据结构。

### 6.1 页面观测对象

`PageObservation`

- `page_name`
- `module_name`
- `screenshot_path`
- `ui_nodes`
- `clickable_nodes`
- `text_nodes`
- `captured_at`

- `attrs`

## 7. 模块化建设要求

必须支持以下模块逐步接入：

- 登录与进入游戏
- 大厅
- 新手流程
- 奖励领取
- 常见弹窗处理
- 战斗开始
- 战斗中
- 战斗结算
- 返回和退出流程

每个模块都应优先通过以下方式接入：

- 配置
- 语义标签补充
- 少量规则

不应通过以下方式接入：

- 大量硬编码 if/else
- 只针对单一项目写死逻辑
- 复制多个项目专属 runner

## 8. 跨项目和跨版本复用策略

### 8.1 复用核心

需要复用的是：

- 页面签名策略
- 页面语义标签
- 控件语义规则
- 通用候选动作模板
- 异常检测器
- 报告结构

### 8.2 项目差异化配置

允许项目自行配置：

- 包名
- Poco 连接参数
- 页面签名关键字
- 模块列表
- 高危动作关键字
- 项目日志关键字
- 战斗和结算关键字

### 8.3 跨版本策略

同一项目不同版本必须优先复用：

- 历史页面签名
- 页面语义映射
- 控件别名
- 已知安全动作
- 已知危险动作

若 UI 发生变化，应先更新映射和语义层，不应直接重写主体逻辑。

## 9. 对 Cursor 的开发约束

后续让 Cursor 继续开发时，必须遵守以下约束：

1. 不要把系统继续实现成“关键词驱动的固定遍历器”。
2. 新功能优先补到观测层、世界模型层、候选动作层、Agent 决策层，不要只在 runner 里堆逻辑。
3. 如果新增页面识别能力，优先建设语义标签和页面模型，不要只增加字符串匹配。
4. 如果新增异常检测能力，必须输出结构化问题对象，而不是只打印日志。
5. 如果新增报告能力，必须服务于“问题定位”和“复现路径”，而不是只增加统计数字。
6. 如果需要 AI 决策，优先采用“系统筛选候选动作 + AI 选择”的模式。
7. 不要假设一次运行就能拿到全部 UI 树，世界模型必须支持持续补全。
8. 所有 Agent 决策必须可追踪，至少保留输入摘要、选择理由、执行结果。
9. 新项目接入优先走配置和语义映射，不要复制一份新的项目专用主流程代码。
10. 如果现有实现与本文档目标冲突，优先记录冲突点，并提出重构方案。

## 10. 建议的开发顺序

严格建议按以下顺序推进：

1. 统一观测数据结构
2. 建立页面快照与状态图存储
3. 建立页面和控件语义化层
4. 建立候选动作生成器
5. 建立 Agent 单步决策协议
6. 建立异常检测器
7. 建立结构化报告生成器
8. 再做长时巡航和多项目复用

不建议一开始就直接：

- 上大模型自由控 UI
- 追求一次采全量 UI
- 追求 24 小时长跑
- 为多个项目堆专用分支逻辑

## 11. 当前阶段的实际落地建议

如果从当前代码基础继续推进，建议下一步优先实现：

1. 新增统一的 `PageObservation` 和 `CandidateAction` 结构。
2. 把页面识别从简单字符串命中升级为“页面签名 + 语义标签”双层机制。
3. 增加控件语义标注流程，让按钮不仅有 `text/name`，还有“作用”。
4. 设计 Agent 的单步输入输出协议，而不是直接在 runner 里写决策。
5. 把异常检测与动作执行回路绑定。
6. 最后再接入 AI 总结报告。

## 12. 最终验收标准

满足以下标准时，才认为方案方向正确：

- AI 能识别当前页面和主要可交互对象
- AI 能在候选动作中做出可解释的下一步选择
- AI 能推进新手流程和基础战斗流程
- 系统能持续补全世界模型
- 崩溃、阻塞、项目报错可稳定产出问题记录
- 数值错误、资源缺失可在典型场景下做初步识别
- 跑测结束后可输出结构化总结报告

## 13. 一句话总结

本项目要做的不是“自动点点点”，而是“能观察、能理解、能决策、能发现问题、能总结的 AI 手游测试 Agent”。
