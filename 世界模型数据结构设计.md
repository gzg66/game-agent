# 世界模型数据结构设计

## 1. 文档目的

本文档定义 AI 手游测试系统中“世界模型”的数据结构设计，用于统一页面建模、控件建模、动作建模、状态图建模、异常建模和记忆召回接口。

后续无论使用内存结构、JSON 文件、SQLite、向量库还是服务端数据库，逻辑层都应优先对齐本文档定义的抽象对象。

## 2. 世界模型的定位

世界模型不是原始 UI 树的简单存档，而是系统对游戏当前和历史交互状态的结构化认知层。

它至少要回答以下问题：

- 当前页面是什么
- 当前有哪些可交互对象
- 这些对象分别可能有什么作用
- 历史上点它们后发生过什么
- 当前页面属于哪个模块
- 这个页面是否危险、是否异常、是否阻塞
- 从当前页面通常如何推进目标

## 3. 设计原则

### 3.1 原始观测与语义建模分层

必须区分：

- 原始观测数据
- 语义理解结果
- 历史统计结果

不要把三者混在一个对象里随意扩展。

### 3.2 页面、控件、动作、转移四类对象分开建模

核心对象最少应包括：

- 页面对象
- 控件对象
- 动作对象
- 状态转移对象

### 3.3 兼容跨版本变化

同一页面在不同版本下可能存在：

- 文案变化
- 控件顺序变化
- 局部节点变化
- 新增弹窗

因此必须同时保存：

- 稳定签名
- 可变特征
- 语义标签

### 3.4 支持持续补全

世界模型必须支持：

- 新页面加入
- 已知页面语义修正
- 控件语义修正
- 新路径加入
- 问题记录追加

## 4. 总体对象关系

建议主要对象包括：

1. `PageObservation`
2. `PageState`
3. `SemanticNode`
4. `CandidateAction`
5. `ActionExecution`
6. `StateTransition`
7. `IssueRecord`
8. `ModuleState`
9. `MemoryHint`

对象关系建议如下：

- `PageObservation` 是一次原始观测
- `PageState` 是对同类页面的归纳状态
- `SemanticNode` 是页面内控件的语义结果
- `CandidateAction` 是基于控件和记忆生成的候选动作
- `ActionExecution` 是一次动作实际执行记录
- `StateTransition` 是页面状态之间的边
- `IssueRecord` 绑定页面、动作和日志
- `ModuleState` 描述当前模块阶段
- `MemoryHint` 用于给 Agent 决策提供召回信息

## 5. 原始观测层对象

## 5.1 `PageObservation`

代表一次页面观测快照。

建议字段：

- `observation_id`
- `session_id`
- `run_id`
- `step_index`
- `page_signature`
- `page_name_raw`
- `screenshot_path`
- `captured_at`
- `dwell_time_ms`
- `ui_tree`
- `root_nodes`
- `clickable_nodes`
- `text_nodes`
- `device_context`
- `log_slice_refs`
- `metadata`

说明：

- `ui_tree` 保留完整原始树
- `clickable_nodes` 与 `text_nodes` 便于快速消费
- `page_signature` 用于与历史页面进行聚类和归并

## 5.2 `DeviceContext`

建议字段：

- `device_uri`
- `package_name`
- `screen_size`
- `orientation`
- `network_state`
- `battery_level`
- `memory_info`

## 5.3 `RuntimeLogSlice`

建议字段：

- `log_slice_id`
- `source`
- `captured_at`
- `lines`
- `keywords_hit`
- `severity`

## 6. 页面语义层对象

## 6.1 `PageState`

表示对一类页面的结构化归纳，而非某一次观测。

建议字段：

- `page_signature`
- `canonical_page_name`
- `page_type`
- `module_name`
- `semantic_tags`
- `version_tags`
- `confidence`
- `root_features`
- `key_texts`
- `key_clickables`
- `common_actions`
- `risk_flags`
- `first_seen_at`
- `last_seen_at`
- `seen_count`

说明：

- `canonical_page_name` 是系统归纳后的稳定名称
- `page_type` 表示页面类别，例如 `dialog`、`battle`、`lobby`
- `semantic_tags` 表示更细粒度语义标签

## 6.2 页面类别建议枚举

建议统一使用有限类别，例如：

- `login`
- `lobby`
- `dialog`
- `module_entry`
- `reward`

- `battle_running`
- `battle_result`
- `guide`

## 6.3 `ModuleState`

描述页面在测试流程中的模块位置。

建议字段：

- `module_name`
- `stage_name`
- `goal_name`

## 12. 存储策略建议

## 12.1 必须持久化的内容

- 页面观测摘要
- 页面语义状态
- 控件语义结果
- 动作执行记录
- 状态转移边
- 异常问题记录

## 12.2 可以按层拆分存储

建议拆成以下几类存储：

1. 原始观测存储
2. 语义状态存储
3. 动作与状态图存储
4. 异常与报告存储

## 12.3 推荐持久化形式

早期阶段建议：

- JSON / JSONL
- SQLite

中期阶段建议：

- SQLite 或 PostgreSQL
- 截图与原始树单独落文件

后期若引入相似页面检索，可增加：

- 向量索引

## 13. 版本与项目隔离策略

所有核心对象建议都带以下维度中的一部分：

- `project_name`
- `package_name`
- `version_tag`
- `engine_type`

原因：

- 支持跨项目复用
- 支持跨版本回放
- 支持针对单项目做语义覆盖

## 14. 页面归并策略建议

不要把每次观测都当成新页面。

建议采用两层机制：

1. 稳定页面签名
2. 页面归并规则

归并时可综合：

- 根节点特征
- 关键文本
- 可点击节点分布
- 页面语义标签

## 15. 开发约束

后续开发世界模型时，必须遵守：

1. 不要把原始 UI 树直接当世界模型。
2. 不要只存页面签名，不存页面语义。
3. 不要只存动作流水，不存状态转移关系。
4. 不要让异常记录脱离页面和动作上下文。
5. 不要把跨版本变化写死在代码里，应优先落到语义映射和别名层。
6. 不要只做内存态结构，关键对象必须可持久化。

## 16. 最小可落地版本建议

如果当前阶段只做 MVP，建议至少先落以下对象：

1. `PageObservation`
2. `PageState`
3. `SemanticNode`
4. `CandidateAction`
5. `ActionExecution`
6. `StateTransition`
7. `IssueRecord`

只要这 7 个对象设计稳定，后续 Agent、报告和异常检测都会更容易扩展。

## 17. 与 Agent 协议的关系

世界模型负责“存储和召回认知”，Agent 协议负责“单步使用这些认知做决策”。

建议搭配文档：

- `AI手游自主游玩测试整体技术方案.md`
- `Agent单步决策协议.md`

## 18. 一句话总结

世界模型不是页面截图和 UI 树的仓库，而是 AI 对“当前处境、可做动作、历史结果、风险与目标”的结构化记忆系统。
